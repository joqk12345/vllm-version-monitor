# Evolution Brief 生成约束

使用 `templates/evolution-brief.md` 生成项目简报时：

1. 只基于输入的官方 Release、官方文档和已合并 PR 证据；每个里程碑必须带链接。
2. 把 `stable`、`post/patch`、`rc/prerelease` 分开陈述，不能把候选版本当稳定版本。
3. 先按技术拐点分为 4–6 个阶段，再写阶段主题；不要机械按 `major.minor` 切段。
4. 每条“核心判断”必须能回溯到至少两个里程碑或明确标记为分析推断。
5. 里程碑表只保留改变架构、能力边界、性能路径或部署形态的版本（8–15 条）。
6. “技术选型启示”应是条件化建议，不应写成无证据的性能排名。
7. 若资料不足，保留模板字段并标记“待人工确认”，不要虚构阶段、日期或成熟度。
